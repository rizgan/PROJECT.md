**मल्टी-एजेंट पाइपलाइन** के लिए एक ओपन फ़ाइल फ़ॉर्मेट।

एक Markdown फ़ाइल पूरे प्रोजेक्ट का वर्णन करती है: कौन से एजेंट चलते हैं, किस क्रम में, किन मॉडलों के साथ, उन्हें क्या करने की अनुमति है, और विफल होने पर क्या होता है। एक ऑर्केस्ट्रेटर फ़ाइल पढ़ता है और पाइपलाइन चलाता है — कोई ग्लू कोड की आवश्यकता नहीं।

यदि [AGENTS.md](https://agents.md) **एक एजेंट** को बताता है कि रिपो में कैसे व्यवहार करना है, तो PROJECT.md **एक ऑर्केस्ट्रेटर** को बताता है कि पूरे प्रोजेक्ट को कैसे चलाना है।

---

## न्यूनतम उदाहरण

```markdown
---
spec_version: 0.1
id: PROJECT-01
name: Hello pipeline
---

## Agents

### writer
wave: 1
Write a one-paragraph summary of: {{ topic }}.

### reviewer
wave: 2
after: writer
Check the summary for factual errors. Return `approved` or `rejected`.
```

यह एक वैध PROJECT.md है। बाकी सब वैकल्पिक है।

---

## क्यों

आज, एक मल्टी-एजेंट पाइपलाइन को परिभाषित करने का मतलब है ऑर्केस्ट्रेशन कोड लिखना। हर नए प्रोजेक्ट का मतलब है नई क्लासेस, नई वायरिंग, नई कॉन्फ़िग फ़ाइलें। पाइपलाइन परिभाषा डेटा है, कोड नहीं — यह एक फ़ाइल में होनी चाहिए।

PROJECT.md वह फ़ाइल है।

---

## डिज़ाइन सिद्धांत

1. **Markdown + YAML फ्रंटमैटर।** कोई नई भाषा सीखने की जरूरत नहीं।
2. **कोर छोटा रहता है।** यदि कोई फीचर 80% पाइपलाइन को नहीं चाहिए, तो वह एक्सटेंशन में जाता है, कोर में नहीं।
3. **घोषणात्मक, अनिवार्य नहीं।** फ़ाइल *क्या* बताती है; ऑर्केस्ट्रेटर *कैसे* तय करता है।
4. **फ्रेमवर्क-स्वतंत्र।** कोई भी ऑर्केस्ट्रेटर इसे लागू कर सकता है।
